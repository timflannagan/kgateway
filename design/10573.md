# EP-10573: AWS Lambda Upstream Integration

* Issue: [#10573](https://github.com/kgateway-dev/kgateway/issues/10573)

## Background

This enhancement proposal outlines the addition of AWS Lambda support to kgateway's Upstream CRD. This feature will enable users to configure AWS Lambda functions as backend services, maintaining feature parity with similar gateway products while providing seamless serverless integration capabilities.

The implementation section explores the use of the upstream Envoy AWS-related filter(s) to support this feature instead of the custom lambda filters that live in the envoy-gloo Envoy distribution.

## Motivation

As serverless architectures become increasingly prevalent, supporting AWS Lambda as a backend service type is crucial for kgateway to remain competitive and provide users with comprehensive cloud integration options. This enhancement will:

- Enable direct integration with AWS Lambda functions from kgateway
- Maintain feature parity with similar gateway products
- Provide a seamless experience for users adopting serverless architectures
- Allow for flexible configuration through the Gateway API's HTTPRoute resources

### Goals

- Extend the Upstream CRD to support AWS Lambda configuration
- Implement a plugin infrastructure for AWS Lambda integration
- Support secure AWS credential management
- Enable Lambda function referencing through backendRef in HTTPRoute

### Non-Goals

- Supporting other serverless platforms (e.g., Azure Functions, Google Cloud Functions)
- Implementing advanced AWS Lambda features beyond basic function invocation

## Implementation Details

### Configuration

The implementation will extend the Upstream CRD with AWS Lambda-specific configuration:

```yaml
apiVersion: gateway.kgateway.dev/v1alpha1
kind: Upstream
metadata:
  name: my-lambda-upstream
spec:
  aws:
    credentials:
      secretRef:
        name: aws-creds
        namespace: default
    region: us-west-2
    lambda:
      functionName: my-function
      qualifier: prod  # Optional: version/alias
```

### Plugin

The AWS Lambda plugin will be implemented in the `internal/kgateway/extensions2/plugins/upstream` directory:

1. Create new plugin package for AWS Lambda
2. Implement necessary interfaces from the plugin framework
3. Add configuration validation
4. Handle AWS credential management
5. Implement Lambda function invocation logic

### Controllers

Updates to existing controllers:

1. Extend Upstream controller to handle AWS Lambda configuration
2. Implement validation webhooks for AWS Lambda upstreams
3. Add status conditions for AWS connectivity and function availability
4. Implement RBAC updates for AWS credential access

### Deployer

Helm chart modifications:

1. Add AWS credential secret template
2. Include example Lambda upstream configurations
3. Update RBAC rules for AWS secret access

### Translator and Proxy Syncer

Implementation details for translation layer:

1. Add Lambda upstream type translation
2. Implement solo.io filter configuration
3. Prepare abstraction layer for future migration to upstream filter
4. Handle function qualifier and version management

## Alternatives

1. Function Reference Approaches:
   - Option 1: Encode directly in Upstream API
     - Pros: Clear ownership, direct configuration
     - Cons: Less flexible, potential duplication

   - Option 2: Use synthetic parameter in backendRef filters
     - Pros: More flexible, reusable
     - Cons: More complex, indirect configuration

2. AWS Credential Management:
   - Option 1: Secret references only
     - Pros: Simple, secure
     - Cons: Limited flexibility

   - Option 2: Multiple credential sources
     - Pros: More flexible
     - Cons: Increased complexity
